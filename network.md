### 一、HTTP 协议

**1. 基本概念**  
- 超文本传输协议，基于请求-响应模型，无状态。
- 默认端口 80，基于 TCP。
- 报文结构：起始行（方法+URL+版本 / 版本+状态码+短语）、头部、空行、正文。

**2. HTTP 方法及幂等性**  
- GET：获取资源，幂等。
- POST：新增数据，非幂等。
- PUT：整体替换，幂等。
- PATCH：部分修改，非幂等。
- DELETE：删除，幂等。
- HEAD / OPTIONS：头信息查询/预检请求。

**3. 常见状态码**  
- 2xx：200 OK；201 Created；204 No Content。
- 3xx：301 永久重定向（换域名）；302 临时重定向；304 Not Modified（缓存）。
- 4xx：400 Bad Request；401 Unauthorized；403 Forbidden；404 Not Found；405 Method Not Allowed。
- 5xx：500 Internal Server Error；502 Bad Gateway；503 Service Unavailable；504 Gateway Timeout。

**4. HTTP 版本差异（面试重点）**  
- **HTTP/1.0**：短连接，每次请求都新建 TCP 连接。
- **HTTP/1.1**：
  - 默认长连接（`Connection: keep-alive`），复用连接减少开销。
  - 管道化（Pipelining）：允许在前一个响应回来前就发下一个请求，但响应必须按顺序返回，易线头阻塞。
  - 缓存策略更丰富（Cache-Control，ETag 等）。
  - Host 头支持一台服务器多域名。
- **HTTP/2.0**：
  - 二进制分帧，多路复用，同域名下所有通信在单一 TCP 连接上并行交错发送。
  - 头部压缩（HPACK），服务端推送。
  - 仍然无法解决 TCP 层面的线头阻塞（丢包时会阻塞整个连接）。
- **HTTP/3.0**：
  - 基于 QUIC（UDP），从传输层解决了线头阻塞。
  - 0-RTT 建连，内置 TLS 1.3。
  - 连接迁移，网络切换时无需重新握手。

（追问：iOS 中 `URLSession` 默认支持 HTTP/1.1 和 HTTP/2，HTTP/3 需在服务端和客户端均开启 NSUrlSession 的 `assumesHTTP3Capable`。）

### 二、HTTPS 与加密

HTTPS = HTTP over SSL/TLS，端口 443。

**1. 对称加密**  
- 同一个密钥进行加密和解密，速度快。
- 算法：AES、ChaCha20、DES（已不安全）。
- 问题：密钥在传输过程中容易被中间人截获。

**2. 非对称加密**  
- 公钥加密、私钥解密，或私钥签名、公钥验签。
- 速度慢，安全性高。
- 算法：RSA、ECC（椭圆曲线）。
- 用于身份认证和安全传输对称密钥。

**3. 混合加密机制**  
实际 HTTPS 中，通信的内容使用**对称加密**，而对称密钥的交换使用**非对称加密**。

**4. 数字证书与 CA**  
- 服务端把公钥、域名等信息提交给 CA，CA 用自己私钥签名生成证书。
- 客户端内置 CA 公钥，验证证书合法性，防止中间人伪造公钥。
- iOS 中 ATS（App Transport Security）强制 HTTPS，可配置证书绑定（SSL Pinning）增强安全性。

**5. TLS 握手过程（以 1.2 为例，经典八股）**  
1. 客户端 -> 服务端：ClientHello（支持的 TLS 版本、密码套件、随机数1）
2. 服务端 -> 客户端：ServerHello（选定的套件、随机数2）、Certificate（证书）、ServerKeyExchange（如需）、ServerHelloDone
3. 客户端：验签证书，生成预主密钥随机数3，用服务端公钥加密发送（ClientKeyExchange）
4. 两端：用随机数1、2、3 生成相同的会话密钥（对称密钥）
5. 客户端：ChangeCipherSpec（通知后续用会话密钥加密）、Finished（握手数据摘要）
6. 服务端：同样方式回复 ChangeCipherSpec 和 Finished
7. 握手完成，开始对称加密传输

（TLS 1.3 精简为 1-RTT，0-RTT 模式可复用之前会话，省去密钥协商步骤。）

### 三、TCP 协议

**1. TCP 特点**  
- 面向连接、可靠、字节流。
- 提供校验和、序列号、确认应答、超时重传、流量控制、拥塞控制。
- 应用于 HTTP、FTP、邮件等。

**2. TCP 报文结构（记住关键字段）**  
源/目标端口、序列号、确认号、数据偏移（头部长度）、标志位（SYN, ACK, FIN, RST, PSH, URG）、窗口大小、校验和、紧急指针。

**3. 三次握手（八股核心）**  
- 第一次：客户端 -> 服务端 SYN=1, seq=x（SYN_SENT）
- 第二次：服务端 -> 客户端 SYN=1, ACK=1, seq=y, ack=x+1（SYN_RCVD）
- 第三次：客户端 -> 服务端 ACK=1, seq=x+1, ack=y+1（ESTABLISHED）
- 为什么不是两次？防止旧连接 SYN 传到服务端造成错误建立；确认双方收发能力。
- 为什么不是四次？服务端可以把 SYN 和 ACK 合并发送，提高效率。

**4. 四次挥手**  
- 第一次：主动方 -> 被动方 FIN=1, seq=u（FIN_WAIT_1）
- 第二次：被动方 -> 主动方 ACK=1, seq=v, ack=u+1（CLOSE_WAIT，主动方进入 FIN_WAIT_2）
- 第三次：被动方 -> 主动方 FIN=1, ACK=1, seq=w, ack=u+1（LAST_ACK）
- 第四次：主动方 -> 被动方 ACK=1, seq=u+1, ack=w+1（TIME_WAIT），等待 2MSL 后进入 CLOSED
- 为什么四次？全双工，双方都可能还有数据要发，被动方收到 FIN 后可能仍需发送剩余数据，故 ACK 和 FIN 分开发送。
- **TIME_WAIT 为什么等待 2MSL？** 确保最后一个 ACK 能重传到被动方（如果丢包被动方会重发 FIN），同时让网络中本连接的延迟报文消失。

**5. 流量控制与拥塞控制**  
- 流量控制：滑动窗口，接收方通过窗口大小告知发送方还能收多少，避免压垮接收端。
- 拥塞控制：慢启动、拥塞避免、快速重传、快速恢复。通过拥塞窗口（cwnd）和慢启动阈值（ssthresh）动态调整发送速率。

### 四、UDP 协议

- 无连接、不可靠、面向报文，发送时一次性发送一个完整的报文。
- 头部只有 8 字节：源端口、目的端口、长度、校验和。
- 优点：低开销、低延迟，无需建立连接，支持广播、多播。
- 应用：音视频通话、直播、DNS、IoT 等。
- QUIC 协议正是基于 UDP 实现了类似 TCP 的可靠传输，并解决了线头阻塞。

**TCP vs UDP 对比：**  
| 特征       | TCP                | UDP             |
|----------|--------------------|-----------------|
| 连接       | 面向连接           | 无连接           |
| 可靠性     | 可靠（确认重传）   | 不可靠           |
| 顺序       | 有序               | 无序             |
| 传输效率   | 较低（握手、状态） | 高               |
| 首部大小   | 20~60 字节          | 8 字节           |
| 适用场景   | 文件传输、网页     | 实时通信、DNS    |

### 五、其他高频相关补充

**1. DNS 解析过程**  
- 浏览器/客户端查缓存，然后递归查询本地 DNS 服务器，本地 DNS 服务器迭代查询根域名服务器 -> 顶级域名服务器 -> 权威域名服务器，返回 IP。
- 优化：HTTPDNS（防劫持，iOS 常用阿里云/腾讯云 HTTPDNS）、DNS over HTTPS/TLS。

**2. Cookie 和 Session**  
- Cookie：客户端存储，自动携带，可被劫持。
- Session：服务端存储，通过 SessionID（通常存于 Cookie）识别。
- iOS 中 `URLSession` 默认管理 Cookie 容器，可设置 `HTTPCookieStorage`。

**3. 从输入 URL 到页面展示（全套链路，面试必考）**  
- URL 输入 -> DNS 解析 -> TCP 三次握手 -> TLS 握手（如果是 HTTPS） -> 发送 HTTP 请求 -> 服务器处理并响应 -> 浏览器/客户端解析渲染。
- iOS 中的网络典型栈：`URLSession` -> `Network.framework` / `CFNetwork` -> 底层 socket。

**4. WebSocket**  
- 全双工通信协议，基于 HTTP 升级握手（101 Switching Protocols），之后为帧通信（opcode 区分文本、二进制、Ping/Pong）。
- iOS 可使用 `URLSessionWebSocketTask` 或第三方库（如 Starscream）。

**5. iOS 网络性能优化点**  
- 合理使用缓存（URLCache, ETag/304）。
- 减少 TLS 握手：保持长连接、设置合理的 `keep-alive` 超时、使用 session 复用。
- 使用 HTTP/2 或 HTTP/3 多路复用。
- 域名收敛？HTTP/2 下反而不需要，多路复用本身解决了连接数限制。
- 弱网优化：超时、重试、预加载、QUIC 的 0-RTT。

### 第一部分：TCP 的三次握手与四次挥手（重点在“为什么”）

这是面试必考且最容易追问细节的地方。

**1. 三次握手：建立连接**
- **核心目标**：**确认双方的收发能力都是正常的**。通信是全双工，你需要确认你能发对方能收，对方能发你也能收。
- **过程拆解（你可以这样讲给面试官听）**：
  - **第一次握手（客户端 -> 服务端：SYN）**
    - 客户端说：“我要连接了，我发的第一个包序号是 `x`。”
    - 服务端收到了。结论：**服务端知道，客户端的发送能力是正常的。**
  - **第二次握手（服务端 -> 客户端：SYN+ACK）**
    - 服务端回答：“好的收到了（`ack = x+1`，表示期待你下一个发 `x+1`），我也同意连接，我发你的第一个包序号是 `y`。”
    - 客户端收到了。结论：**客户端知道，自己的收发能力、服务端的收发能力都正常。** 此时客户端就认为连接已建立，进入 `ESTABLISHED` 状态。
  - **第三次握手（客户端 -> 服务端：ACK）**
    - 客户端回答：“收到你的同意（`ack = y+1`）。”
    - 服务端收到了。结论：**服务端知道了，自己的发送能力正常。** 至此，服务端也进入 `ESTABLISHED` 状态。

- **追问：“为什么不是两次握手？”**
  - **为了防止已失效的连接请求报文段突然又传到了服务端，因而产生错误。**
  - 场景：客户端发的第一个连接请求 `SYN1` 在网络中滞留很久。客户端以为丢了，就又发了个 `SYN2`，双方通过两次挥手就把 `SYN2` 这个连接建立好、通信完并关闭了。这时，那个老旧的 `SYN1` 终于到了服务端。如果只有两次挥手，服务端一收到 `SYN1` 就回 `SYN+ACK` 建立连接并进入 `ESTABLISHED` 状态，但客户端早忘了这事，根本不理。服务端就在那死等，白白浪费资源。

- **追问：“为什么不是四次握手？”**
  - 纯属浪费。服务端的 `SYN`（表示连接请求）和 `ACK`（表示对客户端的应答）在时间上是带在一起的，完全可以合并发送，提高效率。

**2. 四次挥手：拆除连接**
- **核心原因**：**TCP是全双工的，一个方向的连接需要一次FIN和一次ACK来独立关闭。** 关了“我发你”这条通道，不意味着“你发我”这条也关了。
- **过程拆解（假设是客户端主动关闭）**：
  - **第一次挥手（客户端 -> 服务端：FIN）**
    - 客户端：“我没有数据要发给你了，我要关掉我这边的发送通道了。” （进入 `FIN_WAIT_1` 状态）
  - **第二次挥手（服务端 -> 客户端：ACK）**
    - 服务端：“好的，我知道你发完了。”（进入 `CLOSE_WAIT` 状态，客户端收到后进入 `FIN_WAIT_2`）
    - 这个时候，**TCP连接处于“半关闭”状态**：客户端不再发数据，但可以收；服务端还可以继续向客户端发数据（把剩下的发完）。
  - **第三次挥手（服务端 -> 客户端：FIN）**
    - 服务端：“我的数据也发完了，现在我也要关掉我这边的发送通道了。”（进入 `LAST_ACK` 状态）
  - **第四次挥手（客户端 -> 服务端：ACK）**
    - 客户端：“好的，知道了。”（进入 `TIME_WAIT` 状态，等 2MSL 后彻底关闭。服务端收到 ACK 后立即关闭。）

- **追问：“为什么TIME_WAIT状态要等 2MSL（Maximum Segment Lifetime，报文最大生存时间）？”**
  - **原因一：确保最后一个ACK报文能到达服务端。** 如果这个ACK丢了，服务端会超时重发FIN。客户端在这个2MSL时间内，能收到这个重发的FIN，然后重新发一个ACK，重启计时器。
  - **原因二：让本连接持续时间内产生的所有报文从网络中消失。** 这样，下次再建新连接时，不会出现旧连接的“幽灵”报文乱入。

### 第二部分：HTTP版本的演进（核心是理解“阻塞”问题）

你把这些版本演进想象成一个解决拥堵的故事。

- **HTTP/1.0**
  - “过一次手就甩”：每次请求都建一个TCP连接，用完就关。每次都要三次握手和四次挥手，非常慢。

- **HTTP/1.1**
  - **“长连接”**：`Connection: keep-alive`。一个TCP连上了就用着，直到超时或主动关闭，省了重复握手的时间。
  - **“管道化”**：不排队，可以连续发请求。比如我同时要A、B两个图，我连着发两个请求给服务器，不用等A回了再发B。
  - **“线头阻塞 (Head-of-line Blocking, HOL)”**：**这是它的致命伤。** 服务器必须按收到请求的顺序，依次返回响应。如果A的响应特别慢，就算B早处理完了，也得在后面排着，硬等。关键这是发生在**应用层**的阻塞。

- **HTTP/2**
  - **“多路复用”**：这是最大的改进，彻底解决了 HTTP/1.1 的应用层线头阻塞。
    - **分帧**：把请求/响应数据切成小块的“帧（Frame）”。
    - **流**：来自不同请求的帧，在一条 TCP 连接上交错地发送。接收端拿到帧后，根据帧头的“流标识符”，把它们重新组装成完整的响应。
    - **效果**：A 的响应慢没关系，B 的响应帧可以先发、先组合。一条路上跑多路车，谁先到都行。
  - **新瓶颈**：**TCP 层面的线头阻塞**。 HTTP/2 所有流都在同一条 TCP 连接上。TCP 要保证可靠有序。如果底层有一个 TCP 包丢了，那之后所有的包都得在缓冲区等着，直到这个包被重传成功，才能向上层交付。这样一来，上面所有的流（包括 B、C、D）全被卡住了。

- **HTTP/3**
  - **换赛道：基于 QUIC（Quick UDP Internet Connections）协议，用 UDP 而不是 TCP。** 本质上是在应用层用UDP重新实现了一套带多路复用的可靠传输。
  - **如何解决TCP的线头阻塞？** QUIC 的“多路复用”是真正无阻塞的。因为不同“流”的数据被独立处理。如果流1丢了一个包，只需要等流1重传，流2和流3的数据照样可以无损地顺序向上层交付，互相之间不受影响。
  - **其他杀手锏**：
    - **0-RTT 建连**：之前连过的服务器，再次连接时可以加密数据直接就发过去，省去握手轮的来回。
    - **连接迁移**：连接不以（源IP+端口）来标识，而是用 Connection ID。WiFi切4G，IP和端口都变了，但连接ID不变，连接还在，不用重连。

### 第三部分：HTTPS 的加密与TLS握手（把“混合加密”流程刻在脑子里）

- **为什么是“混合”？**
  - **对称加密**（AES等）：又快又安全，但问题在于，通信双方怎么在互联网上安全地传递这“唯一的一把钥匙”？发过去路上就被偷了。
  - **非对称加密**（RSA/ECC）：天然安全（公钥能上锁不能开锁），但速度巨慢。
  - **所以方案呼之欲出**：用非对称加密，安全地传递对称加密的“会话钥匙”。拿到钥匙后，接下来漫长的通信全用对称加密。

- **TLS 1.2 握手过程（关键问：第三个随机数）**
  - 我们一步步看“会话钥匙”是怎么产生的：
    1.  **Client Hello**：客户端给服务器发：“我要通信，我支持的密码套件，这是随机数1号”。
    2.  **Server Hello + 发证书**：服务器回：“咱就用这个套件，这是随机数2号”，并把自己的**数字证书**（内含服务器公钥）发给客户端。
    3.  **客户端验证证书**：客户端用自己系统/App里内置的CA机构的公钥，解开证书的数字签名，确保证书没被篡改，确认公钥真属于这个服务器。
    4.  **生成预主密钥（核心）**：客户端再生成一个**随机数3号（预主密钥 Pre-Master Secret）**，然后它掏出刚刚验证好的**服务器“公钥”**，把随机数3加密，发给服务器。**全世界只有持有对应“私钥”的服务器才能解开。** 此时，两端都有了随机数1、2、3，它们用相同的算法根据这三个数计算出相同的 **“会话密钥”**。
    5.  **切换加密**：客户端发消息：“下一条开始，我发的东西就用会话密钥加密了”。服务器也回同样的话。
    6.  握手完毕，后面的HTTP报文都被对称加密了。

- **TLS 1.3 的精简**
  - 省掉很多，不再支持RSA这类不安全套件。在 Client Hello 时，客户端直接猜一个密钥协商参数，发过去。
  - 如果猜中了（比如和以前连过的服务器重连），可以直接1-RTT就完成握手。重连甚至可以是0-RTT，数据带着业务请求在第一个包就发出去了。

把这些内在逻辑理顺了，这些就不再是枯燥的“八股”，而是你可以在面试中条分缕析讲出来的知识体系。重点准备“三次四次”、“HTTP/2多路复用 vs TCP线头阻塞”、“TLS第三个随机数”这些，基本是必问的。有哪里我讲得不够细致的地方，随时提出来。
