## 摘要
本节课围绕 Objective-C 方法动态解析与消息转发展开，承接上一节 `objc_msgSend` 的快速查找、慢速查找与 `lookupImpOrForward` 流程。当方法在当前类及父类链中都找不到实现时，系统不会立刻崩溃，而是依次提供动态方法解析、快速消息转发、慢速消息转发三次补救机会。课程通过源码阅读、断点调试和 Runtime API 示例，讲解了实例方法、类方法的动态添加实现，以及如何利用消息转发提升程序稳定性和实现 AOP 埋点等能力。


### 亮点
- 🧭 Objective-C 中调用方法的本质是向对象发送消息，编译阶段会将方法调用转换为 `objc_msgSend`。
- 🔍 `objc_msgSend` 会先在当前类的方法缓存和方法列表中查找实现，找不到时继续沿父类链向上查找。
- 🚧 当方法在类和父类中都找不到时，系统不会立即崩溃，而是先进入 `lookupImpOrForward` 后续流程。
- 🛠️ 方法动态解析是第一根“救命稻草”，实例方法会触发 `resolveInstanceMethod:`，类方法会触发 `resolveClassMethod:`。
- 🧩 在动态解析阶段，可以通过 Runtime API 使用 `class_addMethod` 为缺失的 `SEL` 动态绑定一个 `IMP`。
- 🧠 类方法的动态解析比实例方法更特殊，因为类方法存储在元类中，但开发者通常不能直接向元类写代码。
- ⚙️ `resolveClassMethod:` 的设计主要是为了简化类方法动态解析的查找流程，提高系统查找效率。
- 🔁 如果在动态解析中查找或添加方法的逻辑写错，例如在元类中查找不存在的实例方法，可能导致递归调用甚至死循环。
- 🚀 快速消息转发通过 `forwardingTargetForSelector:` 实现，可以把当前对象无法响应的消息转发给另一个能够处理该消息的对象。
- 🐢 慢速消息转发由 `methodSignatureForSelector:` 和 `forwardInvocation:` 共同完成，灵活性比快速转发更高。
- 🧯 动态解析、快速转发和慢速转发共同构成 Objective-C 处理“找不到方法”崩溃前的三次补救机会。
- 📦 将慢速转发逻辑写在 `NSObject` 分类中，可以在一定程度上统一兜底未识别方法，提升线上 App 稳定性。
- 📊 Runtime 与方法交换可以用于 AOP 场景，例如在不修改业务源码的情况下统计页面停留时长。
- 🔄 通过交换 `viewWillAppear:` 等生命周期方法，可以统一采集页面进入和退出时间，从而实现埋点统计。
- ⚠️ 这些兜底机制不应替代正常业务实现，开发调试阶段仍应尽早暴露问题，线上阶段才适合用于稳定性保护。
- 🧪 课程还演示了如何开启 Objective-C 消息日志，观察动态解析与消息转发在崩溃前的调用路径。
- 📚 本节作业是整理一篇关于“方法动态解析与消息转发”的博客，加深对完整消息发送流程的理解。

### 思考 
1. **[方法找不到时，Objective-C 为什么不会马上崩溃？](https://bibigpt.co/search?q=%E6%96%B9%E6%B3%95%E6%89%BE%E4%B8%8D%E5%88%B0%E6%97%B6%EF%BC%8CObjective-C%20%E4%B8%BA%E4%BB%80%E4%B9%88%E4%B8%8D%E4%BC%9A%E9%A9%AC%E4%B8%8A%E5%B4%A9%E6%BA%83%EF%BC%9F)**
  - 因为 Runtime 在真正抛出 `doesNotRecognizeSelector:` 异常之前，会依次尝试方法动态解析、快速消息转发和慢速消息转发。只有这些阶段都没有处理，才会触发找不到方法的崩溃。
- 2. **[动态解析和消息转发有什么区别？](https://bibigpt.co/search?q=%E5%8A%A8%E6%80%81%E8%A7%A3%E6%9E%90%E5%92%8C%E6%B6%88%E6%81%AF%E8%BD%AC%E5%8F%91%E6%9C%89%E4%BB%80%E4%B9%88%E5%8C%BA%E5%88%AB%EF%BC%9F)**
  - 动态解析是在当前类或元类中为缺失方法动态添加实现，本质是补上 `SEL` 对应的 `IMP`。消息转发则是不一定给当前类添加实现，而是把消息交给其他对象，或通过 `NSInvocation` 统一处理，因此慢速转发更灵活。
- 3. **[这些机制在实际项目中有什么用？](https://bibigpt.co/search?q=%E8%BF%99%E4%BA%9B%E6%9C%BA%E5%88%B6%E5%9C%A8%E5%AE%9E%E9%99%85%E9%A1%B9%E7%9B%AE%E4%B8%AD%E6%9C%89%E4%BB%80%E4%B9%88%E7%94%A8%EF%BC%9F)**
  - 它们可以用于线上稳定性兜底、崩溃信息收集、未实现方法监控，也可以结合 Runtime 实现 AOP 埋点，例如统一统计页面停留时间、按钮点击行为或业务流程日志。


### 术语解释
- **objc_msgSend**: Objective-C 方法调用最终会转换成的消息发送函数，负责根据消息接收者和方法名查找具体实现。
- **SEL**: 方法选择器，表示方法名，例如 `test`、`viewWillAppear:`，Runtime 会通过它查找方法实现。
- **IMP**: 方法实现的函数指针，真正执行方法逻辑的地址；动态添加方法时通常需要为某个 `SEL` 绑定一个 `IMP`。
- **方法动态解析**: 当方法找不到时，系统调用 `resolveInstanceMethod:` 或 `resolveClassMethod:`，允许开发者在运行时动态添加方法实现。
- **消息转发**: 当动态解析没有处理方法缺失时，Runtime 提供的后续补救流程，包括 `forwardingTargetForSelector:`、`methodSignatureForSelector:` 和 `forwardInvocation:`。
