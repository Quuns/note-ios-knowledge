## 摘要
本节课主要讲解 iOS 应用程序从点击启动到进入 `main` 函数之前，系统完成的核心加载流程。课程从 Mach-O 文件格式、编译链接过程、静态库与动态库的区别讲起，逐步引出动态链接器 `dyld` 的作用，并结合 `dyld3` 源码分析应用启动时动态库加载、初始化、`libSystem`、`libdispatch`、`libobjc` 以及 `map_images`、`load_images` 等关键流程，为后续类加载与启动优化课程做铺垫。


### 亮点
- 🚀 本节课的核心目标是理解 iOS 应用从启动到 `main` 函数执行之前，系统到底做了哪些准备工作。
- 🧩 Mach-O 是苹果平台上的一种文件格式，可执行文件、目标文件、动态库、静态库以及 bundle 都可能采用这种格式。
- 🛠️ 源代码从 `.h`、`.m`、`.swift` 等文件变成可执行文件，需要经历预编译、编译、汇编和链接等阶段。
- 🔗 静态链接会把多个 `.o` 目标文件以及静态库中的目标文件合并成最终的可执行文件，因此静态库在链接后不再独立存在。
- 📦 动态库可以理解为已经链接完成的镜像文件，运行时需要由动态链接器进行加载和链接。
- 🧠 `dyld` 是 Dynamic Link Editor 的缩写，它负责在应用启动阶段加载依赖的动态库，并完成动态链接相关工作。
- 📲 应用启动时，系统会先调用底层入口流程，创建进程、分配内存、加载 App 的可执行文件，再加载 `dyld`。
- 🔁 `dyld` 加载动态库时是递归过程，因为一个动态库可能还依赖其他动态库，所以需要递归加载完整依赖链。
- 🧭 `dyld` 的主要工作包括加载动态库、执行 rebase 和 binding、初始化动态库与主可执行文件，最后调起 `main` 函数。
- 🧪 通过在 `+load` 方法中打断点并查看堆栈，可以观察到 `+load` 一定发生在 `main` 函数之前。
- 🧱 iOS 15 之后 `dyld` 从 `dyld3` 演进到 `dyld4`，源码结构变化较大，但应用加载的整体理解思路仍然相通。
- 📚 课堂选择 `dyld3` 源码进行分析，是因为它更利于理解完整启动流程，而不是为了逐行读懂所有源码实现。
- 🧬 动态库初始化过程中，`libSystem` 必须较早初始化，因为它会进一步初始化 `libdispatch`，而 `libdispatch` 又会触发 `libobjc` 的初始化。
- 🧵 `libobjc` 初始化时会向 `dyld` 注册回调函数，例如 `map_images`、`load_images` 和 `unmap_image`。
- 🧑‍🏫 `map_images` 和 `load_images` 是后续理解 Objective-C 类加载、分类加载、`+load` 调用时机的关键入口。
- ⚙️ `+load` 方法会影响启动耗时，因为它在 `main` 函数之前执行，里面的耗时操作会直接拖慢冷启动速度。
- 🧊 冷启动是指 App 已经被杀死后重新启动，而热启动通常是指 App 仍在后台存活时再次回到前台。
- 🧰 启动优化并不是脱离加载流程单独讨论，而是建立在理解 `dyld`、动态库初始化和类加载流程的基础之上。

### 思考 
1. **[为什么 `+load` 方法会在 `main` 函数之前执行？](https://bibigpt.co/search?q=%E4%B8%BA%E4%BB%80%E4%B9%88%20%60%2Bload%60%20%E6%96%B9%E6%B3%95%E4%BC%9A%E5%9C%A8%20%60main%60%20%E5%87%BD%E6%95%B0%E4%B9%8B%E5%89%8D%E6%89%A7%E8%A1%8C%EF%BC%9F)**
  - 因为在 `dyld` 初始化动态库和主可执行文件时，`libobjc` 会注册并触发与类加载相关的回调，例如 `load_images`。`+load` 方法就是在这个阶段被调用的，所以它发生在 `main` 函数之前。
- 2. **[`dyld` 在应用启动过程中主要做了什么？](https://bibigpt.co/search?q=%60dyld%60%20%E5%9C%A8%E5%BA%94%E7%94%A8%E5%90%AF%E5%8A%A8%E8%BF%87%E7%A8%8B%E4%B8%AD%E4%B8%BB%E8%A6%81%E5%81%9A%E4%BA%86%E4%BB%80%E4%B9%88%EF%BC%9F)**
  - `dyld` 主要负责加载 App 依赖的动态库，递归处理动态库之间的依赖关系，执行 rebase 和 binding，初始化动态库与主可执行文件，最后把控制权交给 `main` 函数。
- 3. **[为什么启动优化中常说要减少 `+load` 方法的使用？](https://bibigpt.co/search?q=%E4%B8%BA%E4%BB%80%E4%B9%88%E5%90%AF%E5%8A%A8%E4%BC%98%E5%8C%96%E4%B8%AD%E5%B8%B8%E8%AF%B4%E8%A6%81%E5%87%8F%E5%B0%91%20%60%2Bload%60%20%E6%96%B9%E6%B3%95%E7%9A%84%E4%BD%BF%E7%94%A8%EF%BC%9F)**
  - 因为 `+load` 会在 `main` 函数之前执行，它的耗时会直接计入 App 冷启动阶段。如果在 `+load` 中做复杂逻辑、I/O 操作或大量方法交换，就会明显影响启动速度。


### 术语解释
- **Mach-O**: Apple 平台上的二进制文件格式，可用于表示可执行文件、目标文件、动态库、静态库和 bundle 等。
- **dyld**: Dynamic Link Editor，动态链接器，负责在应用启动时加载依赖动态库、完成动态链接并最终调用 `main` 函数。
- **静态链接**: 链接器把多个 `.o` 文件和静态库内容合并到最终可执行文件中的过程，完成后静态库不再作为独立文件存在。
- **动态库 / Image**: 动态库是运行时加载和链接的二进制镜像文件，`image` 可泛指可执行文件、动态库或 bundle 这类 Mach-O 镜像。
- **map_images / load_images**: `libobjc` 注册给 `dyld` 的关键回调函数，和 Objective-C 类加载、分类处理以及 `+load` 方法调用密切相关。
