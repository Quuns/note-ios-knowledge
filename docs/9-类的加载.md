## 摘要
本节课围绕 Objective-C 类加载流程展开，重点从 dyld3/dyld4 与 Runtime 的协作关系讲起，分析 `_objc_init` 初始化阶段、环境变量调试能力、`load_images` 与 `map_images` 回调的作用，并进一步追踪 `read_images`、`readClass`、非懒加载类初始化以及 `class_ro_t`、`class_rw_t` 数据结构的生成时机。


### 亮点
- 🧭 本节课先回顾了 App 启动时 dyld 加载可执行文件、动态库并最终调用 `main` 函数的整体流程。
- 🧩 dyld 在初始化 `libSystem`、`libdispatch` 和 `libobjc` 时，会让 Objective-C Runtime 参与类、分类和方法等元数据的处理。
- 🔁 `dyld_objc_notify_register` 会向 dyld 注册三个关键回调函数：`map_images`、`load_images` 和 `unmap_image`。
- 🧠 dyld3 和 dyld4 的源码封装方式不同，但它们与 Objective-C Runtime 协作的核心流程基本一致。
- ⚙️ `_objc_init` 不只注册回调函数，还会先完成环境变量、TLS、静态构造函数、Runtime 表、异常系统和缓存系统等初始化。
- 🧪 通过 Objective-C 环境变量可以观察 Runtime 行为，例如打印已加载镜像、打印实现了 `+load` 的类，以及关闭 `non-pointer isa` 优化。
- 🧱 `non-pointer isa` 是 Runtime 对对象 `isa` 指针的优化方式，它可以在指针中存储额外状态信息，从而减少内存浪费。
- 🧵 TLS 是线程局部存储，Runtime 会为线程准备相关析构函数，后续递归锁等机制也可能依赖 TLS 保存线程私有数据。
- 🚀 C/C++ 静态构造函数会在特定初始化阶段执行，它适合完成静态数据初始化或只需执行一次的准备操作。
- 📦 `load_images` 的核心职责是查找并调用所有实现了 `+load` 方法的类和分类。
- 👨‍👩‍👧 `+load` 方法的调用顺序通常是父类优先于子类，类优先于分类；多个分类之间则与编译顺序有关。
- 🔒 Runtime 在查找和调用 `+load` 方法时会加锁，因此 `+load` 调用过程具备线程安全特征。
- 🗺️ `map_images` 会参与镜像映射后的 Runtime 初始化处理，包括读取 Mach-O 中的类、协议、选择子和分类等信息。
- 🧰 `read_images` 是类加载流程中的关键函数，它会处理选择子注册、类表注册、协议修复、指针修复和非懒加载类初始化等任务。
- 🎯 ASLR 会导致镜像每次启动时的虚拟地址不固定，因此 dyld 需要通过 rebase 和 bind 等步骤修复资源指针。
- 🧬 `readClass` 主要负责读取已经分配内存的类和元类，并将它们加入 Runtime 的类表中。
- 🏗️ 非懒加载类通常指实现了 `+load` 方法的类，它会在 Runtime 初始化阶段被提前 realize。
- 🧾 `realizeClassWithoutSwift` 是类真正完成初始化的重要函数，其中会为 `class_rw_t` 开辟空间并关联编译期生成的 `class_ro_t` 数据。
- 📚 `class_ro_t` 保存编译期确定的只读信息，例如成员变量、方法列表和属性等；`class_rw_t` 保存运行期可变信息。
- 🔍 本节课最后通过断点和 LLDB 验证了 `class_rw_t` 的赋值时机，并说明懒加载类与分类加载会在后续课程继续展开。

### 思考 
1. **[`+load` 方法为什么不需要手动调用 `super load`？](https://bibigpt.co/search?q=%60%2Bload%60%20%E6%96%B9%E6%B3%95%E4%B8%BA%E4%BB%80%E4%B9%88%E4%B8%8D%E9%9C%80%E8%A6%81%E6%89%8B%E5%8A%A8%E8%B0%83%E7%94%A8%20%60super%20load%60%EF%BC%9F)**
  - 因为 Runtime 在查找 `+load` 方法时会递归处理父类，父类的 `+load` 会被系统自动调用。如果子类里再手动调用 `[super load]`，可能导致父类 `+load` 被执行两次。
- 2. **[什么是懒加载类和非懒加载类？](https://bibigpt.co/search?q=%E4%BB%80%E4%B9%88%E6%98%AF%E6%87%92%E5%8A%A0%E8%BD%BD%E7%B1%BB%E5%92%8C%E9%9D%9E%E6%87%92%E5%8A%A0%E8%BD%BD%E7%B1%BB%EF%BC%9F)**
  - 在本节课的语境中，实现了 `+load` 方法的类会被视为非懒加载类，它会在 Runtime 初始化阶段提前处理；没有实现 `+load` 的类通常属于懒加载类，会在后续真正使用时再完成初始化。
- 3. **[`class_ro_t` 和 `class_rw_t` 的区别是什么？](https://bibigpt.co/search?q=%60class_ro_t%60%20%E5%92%8C%20%60class_rw_t%60%20%E7%9A%84%E5%8C%BA%E5%88%AB%E6%98%AF%E4%BB%80%E4%B9%88%EF%BC%9F)**
  - `class_ro_t` 是编译期生成的只读数据，包含方法、属性、成员变量等静态信息；`class_rw_t` 是运行期生成的可变数据结构，会承载 Runtime 运行过程中需要修改或扩展的信息。


### 术语解释
- **dyld**: Apple 系统中的动态链接器，负责加载 App 可执行文件和动态库，并完成动态链接、符号绑定和初始化调用。
- **`_objc_init`**: Objective-C Runtime 的核心初始化入口，会初始化环境变量、线程局部存储、异常系统、缓存系统，并向 dyld 注册 Runtime 回调函数。
- **`load_images`**: Runtime 注册给 dyld 的回调之一，主要负责发现并调用类和分类中的 `+load` 方法。
- **`map_images` / `read_images`**: `map_images` 在镜像加载后触发，内部会调用 `read_images` 读取 Mach-O 中的类、协议、选择子等 Runtime 元数据，并进行修复和注册。
- **`class_ro_t` 与 `class_rw_t`**: Objective-C 类结构中的两类核心数据，前者保存编译期只读信息，后者保存运行期可变信息，是理解类加载和方法列表管理的重要基础。
