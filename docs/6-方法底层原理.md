## 摘要
本节课围绕 Objective-C 方法调用的底层原理展开，重点讲解 `objc_msgSend`、方法缓存 `cache_t`、`super` 关键字、快速查找与慢速查找流程。课程通过源码、汇编调试和 Demo 验证，说明 Objective-C 的动态性来自 Runtime，方法调用本质是“消息发送”，并且方法会优先从缓存中查找，未命中后再进入方法列表和父类链查找，最终可能进入消息转发。


### 亮点
- 🚀 Objective-C 是一门动态语言，很多调用关系不是在编译期完全确定，而是在运行时通过 Runtime 决定。
- 🧠 Runtime 的核心知识可以概括为两部分：类与对象的动态配置，以及消息发送与消息转发机制。
- 📩 普通 Objective-C 方法调用在编译后会被转换成 `objc_msgSend` 函数调用。
- 🎯 `objc_msgSend` 默认至少包含两个参数，第一个是消息接收者 `receiver`，第二个是方法选择器 `SEL`。
- 🧩 如果原方法本身带有参数，这些参数会追加在 `receiver` 和 `SEL` 后面传入 `objc_msgSend`。
- 🛠️ 在 Xcode 12 之后，手动调用 `objc_msgSend` 时通常需要显式进行函数指针类型强转，否则可能出现参数过多等编译错误。
- 🔍 `super` 并不代表消息接收者变成了父类对象，消息接收者仍然是 `self`，只是方法查找的起点从父类开始。
- 🧱 `objc_msgSendSuper` 的核心参数是 `objc_super` 结构体，其中包含 `receiver` 和 `super_class`。
- ⚠️ 如果错误地把 `objc_msgSendSuper` 的 `super_class` 设置为当前类，可能导致方法反复查找当前类并形成死循环。
- 💥 如果把 `super_class` 设置得过高，例如直接跳到 `NSObject`，而目标方法不存在，就可能触发崩溃。
- ⚡ `objc_msgSend` 使用汇编实现，主要原因是性能更高，可以减少局部变量拷贝，并直接利用寄存器传递参数。
- 🧭 方法快速查找流程会先判断 `receiver` 是否存在，再通过 `isa` 找到类对象，然后从类对象的缓存中查找目标方法。
- 🗂️ 方法缓存中存储的是 `bucket`，每个 `bucket` 关联 `SEL` 和 `IMP`，缓存命中后会直接调用对应的 `IMP`。
- 🐢 如果快速查找没有命中缓存，Runtime 会进入 `objc_msgSend_uncached`，进一步调用 `lookUpImpOrForward` 进行慢速查找。
- 🔁 慢速查找会先再次检查缓存，以避免多线程场景下其他线程刚好完成缓存填充。
- 📚 当前类的方法列表查找使用二分查找算法，因此方法列表需要具备有序性。
- 🧮 二分查找通过不断取中间位置缩小范围，比线性遍历更高效。
- 🧷 分类方法会排在方法列表更靠前的位置，因此当分类和主类存在同名方法时，通常会优先调用分类方法。
- 🧬 如果当前类方法列表中找不到目标方法，Runtime 会沿着父类链继续查找父类缓存和父类方法列表。
- 🧯 如果一直查找到 `NSObject` 仍然没有找到方法，Runtime 才会进入消息转发流程。
- 🗃️ 慢速查找到方法后，会把对应的 `IMP` 插入缓存，便于后续调用走快速路径。
- 🧪 子类调用继承自父类的方法时，方法实现可能来自父类，但缓存通常会与当前消息接收者对应的类有关。
- 🏗️ 类方法和实例方法在底层查找机制上没有本质区别，区别主要在于它们存储在类对象或元类对象的方法列表中。
- 🔄 `NSObject` 分类中的实例方法，在特定类对象调用场景下也可能被查找到，这体现了类、元类和继承链的统一消息机制。

### 思考 
1. **[`super` 调用方法时，消息接收者到底是谁？](https://bibigpt.co/search?q=%60super%60%20%E8%B0%83%E7%94%A8%E6%96%B9%E6%B3%95%E6%97%B6%EF%BC%8C%E6%B6%88%E6%81%AF%E6%8E%A5%E6%94%B6%E8%80%85%E5%88%B0%E5%BA%95%E6%98%AF%E8%B0%81%EF%BC%9F)**
  - 消息接收者仍然是当前对象，也就是 `self`。`super` 只是改变了方法查找的起点，让 Runtime 从父类开始查找方法实现，而不是从当前类开始。
- 2. **[`objc_msgSend` 为什么要用汇编实现？](https://bibigpt.co/search?q=%60objc_msgSend%60%20%E4%B8%BA%E4%BB%80%E4%B9%88%E8%A6%81%E7%94%A8%E6%B1%87%E7%BC%96%E5%AE%9E%E7%8E%B0%EF%BC%9F)**
  - 主要是为了性能。汇编可以更直接地使用寄存器传参，减少不必要的变量拷贝和调用开销，因此更适合 `objc_msgSend` 这种极高频调用的底层函数。
- 3. **[方法查找时，缓存没有命中会发生什么？](https://bibigpt.co/search?q=%E6%96%B9%E6%B3%95%E6%9F%A5%E6%89%BE%E6%97%B6%EF%BC%8C%E7%BC%93%E5%AD%98%E6%B2%A1%E6%9C%89%E5%91%BD%E4%B8%AD%E4%BC%9A%E5%8F%91%E7%94%9F%E4%BB%80%E4%B9%88%EF%BC%9F)**
  - Runtime 会进入慢速查找流程，先在当前类的方法列表中查找，如果找不到再沿父类链继续查找。找到后会缓存对应的 `IMP`，如果最终仍然找不到，则进入消息转发流程。


### 术语解释
- **Runtime**: Objective-C 的运行时系统，提供动态创建类、添加方法、修改属性、消息发送、消息转发等能力，是 Objective-C 动态性的核心。
- **objc_msgSend**: Objective-C 方法调用编译后的底层函数，本质上根据消息接收者和方法选择器查找并执行对应的 `IMP`。
- **SEL**: 方法选择器，表示方法名，例如 `@selector(study)`，Runtime 通过它在缓存或方法列表中匹配目标方法。
- **IMP**: 方法实现的函数指针，找到 `IMP` 后就可以真正执行对应的方法逻辑。
- **cache_t**: 类对象中的方法缓存结构，用于缓存已经调用过的方法，提高后续方法调用速度。
