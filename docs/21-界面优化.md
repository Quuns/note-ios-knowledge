## 摘要
- 本节课先简要穿插了 WWDC 中关于 iOS 锁屏、通知、Live Activities、Messages、Dictation、Live Text、Wallet 等功能的介绍，随后正式进入课程主题：从上一节“界面优化”的边界谈起，进一步讲解项目架构设计。课程重点围绕“高内聚、低耦合”展开，通过 MVC 的问题、轻量化 ViewController、MVP 面向协议编程、MVVM 双向绑定、适配器模式与 Context 中介层设计，帮助大家理解架构选型应服务于项目维护、迭代和复杂业务治理，而不是盲目追求某一种模式。


### 亮点
* 🧭 架构设计的核心目标不是追求某个固定模式，而是让项目达到“高内聚、低耦合”的效果。
* 🧩 所谓高内聚、低耦合，可以简单理解为“谁的事情谁做”，避免 View、Model、Controller 之间职责混乱。
* ⚙️ 界面优化并不是所有页面都必须做，应该结合项目规模、性能瓶颈、开发周期和团队能力来决定优化程度。
* 🚦 如果 App 已经稳定运行且没有明显卡顿，就没有必要为了炫技而引入复杂的下采样、预解码等优化手段。
* 🏗️ 传统 MVC 在小项目中清晰直接，但在复杂项目中容易导致 ViewController 臃肿、代理泛滥和数据修改位置难以追踪。
* 📦 轻量化 ViewController 的关键是让 VC 只负责建立依赖关系，而不是承担 UI 布局、数据加载、列表代理等所有工作。
* 🧱 数据提供层、UI 处理层和视图接管层可以从 VC 中拆分出去，让 Controller 的生命周期和调度逻辑更加清晰。
* 🔁 当 Cell 内部直接修改 Model 时，会造成 UI 与 Model 的强耦合，后续排查数据问题会非常困难。
* 📡 MVP 的价值在于通过协议和代理建立通信，让业务变更通过接口驱动代码，从而让开发思路和维护路径更清晰。
* 🔄 MVVM 的核心是双向绑定，适合局部或中小型场景，但在复杂项目中过多 Block、RAC 信号和嵌套回调会增加调试成本。
* 🧠 架构选型没有绝对最优，代码少也不代表架构好，真正重要的是它是否适合当前项目的业务复杂度和团队维护方式。
* 🧰 适配器模式可以处理多 Cell、多类型列表和局部差异化业务，让基础逻辑复用，具体业务按需适配。
* 🧷 Context 设计相当于一个中介者或百宝箱，用来集中保存 View、Presenter、Adapter、VC 等上下文信息，减少层层传递。
* 🌲 通过响应链和关联对象，可以让复杂子视图也方便地获取 Context，从而降低深层视图通信中的胶水代码。
* 📚 课程最后强调，想成为架构师必须重视设计思想、设计模式和项目全局视角，而不仅仅是记住某种代码写法。

### 思考
1. [什么情况下才需要做界面优化？](https://bibigpt.co/search?q=%E4%BB%80%E4%B9%88%E6%83%85%E5%86%B5%E4%B8%8B%E6%89%8D%E9%9C%80%E8%A6%81%E5%81%9A%E7%95%8C%E9%9D%A2%E4%BC%98%E5%8C%96%EF%BC%9F)
* 当页面已经出现明显卡顿、内存压力、图片加载过重、列表滚动不流畅等问题时，才需要针对性优化。优化应该结合项目阶段、开发成本和性能收益，不应该所有页面都机械套用复杂方案。
1. [MVP 和 MVVM 应该如何选择？](https://bibigpt.co/search?q=MVP%20%E5%92%8C%20MVVM%20%E5%BA%94%E8%AF%A5%E5%A6%82%E4%BD%95%E9%80%89%E6%8B%A9%EF%BC%9F)
* MVP 更适合强调协议、职责分离和可维护性的复杂业务场景；MVVM 适合需要双向绑定、局部状态同步或中小型模块的场景。如果项目层级很深、数据传递复杂，过度使用 MVVM 和 Block/RAC 可能增加调试难度。
1. [为什么要设计 Context 层？](https://bibigpt.co/search?q=%E4%B8%BA%E4%BB%80%E4%B9%88%E8%A6%81%E8%AE%BE%E8%AE%A1%20Context%20%E5%B1%82%EF%BC%9F)
* Context 可以作为中介层集中管理当前模块所需的 View、Presenter、Adapter、VC 等对象，避免子视图和业务层之间层层传递数据，也可以减少复杂页面中的胶水代码和耦合关系。


### 术语解释
* **高内聚低耦合**: 架构设计的核心原则，表示模块内部职责集中，模块之间依赖尽量少，方便维护、扩展和排查问题。
* **MVC**: Model-View-Controller 架构模式，常见于 iOS 开发；优点是直观，缺点是在复杂项目中容易让 ViewController 过于臃肿。
* **MVP**: Model-View-Presenter 架构模式，课程中重点强调通过协议和代理进行通信，让 Presenter 负责协调业务逻辑，降低 View 与 Model 的直接耦合。
* **MVVM**: Model-View-ViewModel 架构模式，核心是通过 ViewModel 实现数据与 UI 的双向绑定，常与 RAC、KVO、Block 等机制结合使用。
* **适配器模式**: 一种设计模式，用于在基础逻辑相同但局部业务不同的情况下进行适配，例如多类型 Cell、多种列表展示和不同业务数据源处理。

### [39:07](https://bibigpt.co/content/947140f3-b836-4d57-a135-f635c5d7d1a7?t=2347.000) - 🧱 轻量化ViewController设计

![章节截图 39:07](https://asset.localhost/C%3A%5CUsers%5CADMIN%5CAppData%5CLocal%5Cco.bibigpt.desktop%5Cscreenshots%5Cvideo_9584123662585615549%5Cscreenshot_2347.0s.jpg)

为缓解VC臃肿，讲师展示了如何剥离冗余代码。将数据请求和数据源代理独立到Presenter和自定义的DataSource中，视图布局代码迁移至独立的KCView，通过伪代码`self.view = self.KCView`实现视图接管。这使得VC只保留生命周期和依赖建立的角色，代码从数千行精简到三十余行。该设计遵循单一职责，让VC回归“产品经理”的协调角色，不再直接操控视图构建和数据细节，为后续架构演进打下基础。

### [59:14](https://bibigpt.co/content/947140f3-b836-4d57-a135-f635c5d7d1a7?t=3554.000) - 📡 MVP与面向协议编程

![章节截图 59:14](https://asset.localhost/C%3A%5CUsers%5CADMIN%5CAppData%5CLocal%5Cco.bibigpt.desktop%5Cscreenshots%5Cvideo_9584123662585615549%5Cscreenshot_3554.0s.jpg)

为彻底解耦视图与模型，讲师引入MVP架构，核心是面向协议编程。通过定义代理协议，将视图的点击事件通过协议方法传递给Presenter，由Presenter修改模型数据，再反向通知视图更新。演示了代理的设置和三部曲，并深入讲解了如何通过`weak`、`strong`修饰符打破循环引用。MVP使代码逻辑清晰，接口驱动开发，新人只需遵循协议即可快速上手。相比MVC，MVP将数据变更集中管理，调试更方便，且避免了数据过度暴露，安全性更高。

### [01:29:23](https://bibigpt.co/content/947140f3-b836-4d57-a135-f635c5d7d1a7?t=5363.000) - 🔁 MVVM与双向绑定对比

![章节截图 01:29:23](https://asset.localhost/C%3A%5CUsers%5CADMIN%5CAppData%5CLocal%5Cco.bibigpt.desktop%5Cscreenshots%5Cvideo_9584123662585615549%5Cscreenshot_5363.0s.jpg)

应学员要求，讲师快速展示了MVVM架构，利用ReactiveCocoa或KVO实现双向绑定。通过一个支付宝示例，点击列表项能同步刷新下方内容，阐明了UI与Model的相互更新。然而，讲师指出MVVM在大型项目中存在Block嵌套过深、调试困难和数据暴露风险。相比之下，MVP的协议回调更具可维护性。因此，架构选型需结合项目复杂度，小型项目或局部功能可用MVVM，大型、多层次项目更适合MVP或后续介绍的更高级模式。

### [01:46:40](https://bibigpt.co/content/947140f3-b836-4d57-a135-f635c5d7d1a7?t=6400.000) - 🔧 适配器模式与复杂UI处理

![章节截图 01:46:40](https://asset.localhost/C%3A%5CUsers%5CADMIN%5CAppData%5CLocal%5Cco.bibigpt.desktop%5Cscreenshots%5Cvideo_9584123662585615549%5Cscreenshot_6400.0s.jpg)

针对直播项目中多种Cell的复杂列表，讲师提出了适配器模式。通过继承BaseAdapter并创建HomeAdapter，将不同Cell的数据源和交互逻辑分发到各自的适配器中，实现差异化处理。适配器作为“协调者”，根据模型类型选择对应的渲染方式，避免了巨型DataSource的产生。该方法使列表代码高度可复用，新增Cell只需扩展适配器，符合开闭原则，显著降低了复杂UI的维护成本。

### [01:57:24](https://bibigpt.co/content/947140f3-b836-4d57-a135-f635c5d7d1a7?t=7044.000) - 🧠 Context设计模式与架构优化

![章节截图 01:57:24](https://asset.localhost/C%3A%5CUsers%5CADMIN%5CAppData%5CLocal%5Cco.bibigpt.desktop%5Cscreenshots%5Cvideo_9584123662585615549%5Cscreenshot_7044.0s.jpg)

为解决MVP中视图与控制器通信仍需胶水代码的问题，讲师设计了Context中介者模式。Context作为百宝箱，持有Presenter、View、Adapter等引用，各组件通过响应链和关联对象获取Context，实现任意层级的直接通信。详细讲解了基于`NSObject`分类和`objc_setAssociatedObject`的实现，以及如何通过weak-root强引用打破循环引用。接着，利用响应链递归查找父视图的Context，使得子视图能轻易触发父级操作，无需显式传递。该模式大幅减少了代理和通知，让代码更聚焦业务逻辑，是高级架构设计的精髓。
