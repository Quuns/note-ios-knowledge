## 摘要
- 本期课程围绕 iOS 项目中的组件化架构展开，重点讲解为什么需要组件化、如何拆分模块、如何使用 CocoaPods 创建本地组件、如何处理三方库与本地库依赖、资源文件加载问题，以及组件之间通信的核心方案。课程最后重点分析了 CTMediator 与 BeeHive 两类解耦思想，强调组件化不是万能架构，而是项目复杂度增长后的工程化选择。


### 亮点
* 🧩 组件化的核心价值在于降低模块之间的耦合，让大型项目在多人协作时更容易维护和扩展。
* 👥 当项目页面和业务越来越多、开发人员越来越多时，直接交叉调用会让沟通成本和维护成本迅速上升。
* 🔁 可复用能力是组件化的重要目标，通用功能、基础工具、网络层、数据库层和公共 UI 都适合抽离成独立模块。
* 🧪 组件化可以降低单元测试成本，因为每个模块都能独立测试，问题定位也会更加清晰。
* ⚠️ 并不是所有项目都需要组件化，如果项目很小、人员很少、模块耦合很低，强行组件化反而会增加复杂度。
* 🏗️ 组件拆分的颗粒度非常关键，拆得太粗会耦合严重，拆得太细又会导致管理成本、学习成本和维护成本上升。
* 📚 推荐将组件体系分为基础模块层、通用模块层和业务模块层，并遵循上层依赖下层、下层尽量不依赖上层的原则。
* ⬇️ 横向依赖应该尽量下沉为公共组件，例如多个页面都需要视频播放能力时，应抽离成独立的视频播放模块。
* 🛠️ 使用 CocoaPods 创建组件时，可以通过 `pod lib create` 生成组件模板，并在 `Classes` 目录中编写组件代码。
* 📦 三方库依赖应写入组件的 `.podspec` 文件中，例如通过 `s.dependency` 声明 AFNetworking、SDWebImage 等依赖。
* 🔗 本地组件之间的依赖需要配置搜索路径或本地路径，否则编译时可能出现头文件找不到、模块找不到等问题。
* 🖼️ 资源文件不能只放进组件目录，还需要在 `.podspec` 中配置资源路径，并通过指定 Bundle 的方式加载图片、JSON、XIB 等资源。
* 🚦 组件化最难的问题是组件通信，常见方案包括 URL Router、Target-Action 中介者模式，以及 Protocol-Class 服务绑定模式。
* 🧭 CTMediator 的思想是通过中间层统一转发调用，让调用方不直接依赖目标模块，从而降低模块之间的直接耦合。
* 🐝 BeeHive 的思想是通过协议与服务绑定来完成模块通信，同时支持系统事件、通用事件和自定义事件的分发。
* 🧠 架构设计的本质不是炫技，而是让代码职责更清晰，遵循高内聚、低耦合和“谁的事情谁做”的原则。

### 思考
1. [什么样的项目适合做组件化？](https://bibigpt.co/search?q=%E4%BB%80%E4%B9%88%E6%A0%B7%E7%9A%84%E9%A1%B9%E7%9B%AE%E9%80%82%E5%90%88%E5%81%9A%E7%BB%84%E4%BB%B6%E5%8C%96%EF%BC%9F)
* 当项目业务复杂、页面模块多、多人协作明显、公共能力复用频繁，并且测试和维护成本逐渐升高时，比较适合引入组件化。如果项目只有一两个人维护、功能简单、耦合关系很少，就没有必要强行组件化。
1. [组件之间应该如何通信，才能避免强耦合？](https://bibigpt.co/search?q=%E7%BB%84%E4%BB%B6%E4%B9%8B%E9%97%B4%E5%BA%94%E8%AF%A5%E5%A6%82%E4%BD%95%E9%80%9A%E4%BF%A1%EF%BC%8C%E6%89%8D%E8%83%BD%E9%81%BF%E5%85%8D%E5%BC%BA%E8%80%A6%E5%90%88%EF%BC%9F)
* 可以使用三类常见方案：URL Router 通过路由字符串定位目标；CTMediator 通过 Target-Action 中介层转发调用；BeeHive 通过 Protocol-Class 或服务注册机制进行解耦。核心目标都是避免模块之间直接 `import` 和直接创建对方对象。
1. [为什么资源文件在组件中经常加载失败？](https://bibigpt.co/search?q=%E4%B8%BA%E4%BB%80%E4%B9%88%E8%B5%84%E6%BA%90%E6%96%87%E4%BB%B6%E5%9C%A8%E7%BB%84%E4%BB%B6%E4%B8%AD%E7%BB%8F%E5%B8%B8%E5%8A%A0%E8%BD%BD%E5%A4%B1%E8%B4%A5%EF%BC%9F)
* 因为组件中的图片、JSON、XIB 等资源通常不在主工程 Bundle 中，不能直接按主工程方式加载。需要在 `.podspec` 中声明资源文件路径，并在代码中通过对应的 Bundle 路径加载资源。


### 术语解释
* **组件化**: 将大型项目按照业务、通用能力和基础能力拆分成多个独立模块，每个模块可以独立开发、测试、维护和复用。
* **CocoaPods / Podspec**: CocoaPods 是 iOS 常用依赖管理工具，Podspec 文件用于描述组件名称、版本、源码地址、依赖库、资源文件等配置信息。
* **CTMediator**: 一种基于 Target-Action 思想的组件通信方案，通过中间调度层转发调用，避免业务模块之间直接依赖。
* **BeeHive**: 一种偏向服务注册和协议绑定的组件化框架，常用于模块生命周期管理、系统事件分发和模块间服务调用。
* **Bundle 资源加载**: 在组件化中，资源文件通常被打包到独立 Bundle，需要通过指定 Bundle 路径加载，而不是默认从主工程资源中读取。
