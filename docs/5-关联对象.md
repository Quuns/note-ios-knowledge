## 摘要
本节课主要围绕 Objective-C Runtime 中 `objc_class` 的 `cache_t` 方法缓存展开，老师从源码、LLDB 调试、内存平移、哈希表插入、缓存扩容规则以及真机验证等角度，系统解释了方法缓存中到底存储什么、如何存储、何时扩容，以及不同架构下扩容策略的差异。


### 亮点
- 🎬 课程正式从打开源码开始，老师说明本节课聚焦 `objc_class` 结构体中尚未讲解的 `cache_t` 字段，也就是方法缓存机制。[33:17]
- 🧠 `cache_t` 从字面上是缓存，但课程重点在于通过源码和调试确认它具体缓存的是 Objective-C 方法相关信息。[34:18]
- 🔍 通过 LLDB 打印类对象地址，并基于 `objc_class` 内存布局平移 16 个字节，可以定位到 `cache_t` 的内存位置。[36:13]
- 🧩 老师进入 `cache_t` 源码后指出，仅看成员变量如 `bucketAndMaybeMask`、`maybeMask`、`occupied` 等，很难直接判断缓存内容。[38:03]
- 🛠️ 通过分析 `cache_t::insert` 方法可以发现，插入参数包含 `SEL` 和 `IMP`，因此方法缓存保存的是方法名和方法实现地址。[39:43]
- 🪣 `bucket_t` 是真正承载缓存条目的结构，内部保存 `SEL` 与 `IMP`，可以把它理解为一个哈希表或散列表中的桶。[41:31]
- 🔐 `bucket_t::set` 方法会把新的 `IMP` 和 `SEL` 写入内存，其中还涉及 `IMP` 编码和原子操作，以保证写入过程的安全性。[42:37]


#️⃣ 方法缓存的插入位置由 `cache_hash(SEL, mask)` 计算得出，哈希结果不会超过桶数组长度减一。[52:19]
- ⚠️ 哈希表可能出现哈希冲突，Runtime 会通过 `cache_next` 继续寻找下一个可用位置来存储方法缓存。[58:07]
- 🧪 老师通过 LLDB 调用 `buckets()` 方法读取 `bucket_t`，验证缓存中确实存在 `class`、`respondsToSelector:` 等方法条目。[01:07:03]
- 🤔 课程中特意引出一个现象：调用 `method1` 后反而找不到该方法缓存，并且原有的 `respondsToSelector:` 也消失，这是为了引出缓存扩容问题。[01:18:52]
- 🧱 `cache_t` 本身大小固定为 16 字节，其中关键成员保存的是指向 `bucket_t` 桶数组的地址，而不是直接存储所有方法。[01:23:02]
- 📐 在 `arm64` 架构下，方法缓存桶的初始长度为 2；在 `x86_64` 架构下，桶的初始长度为 4。[01:45:18]
- 📈 `x86_64` 架构下，当缓存数量超过桶长度的四分之三时会进行两倍扩容；`arm64` 架构下通常以八分之七为阈值，但桶长度小于等于 8 时会等满后再扩容。[02:02:18]
- 🧹 扩容时 Runtime 会重新开辟新的桶数组，并释放旧桶，因此旧缓存条目不会被迁移，这解释了为什么某些已调用方法在缓存中消失。[02:00:51]
- 🧾 `respondsToSelector:` 和 `class` 方法可能是 LLDB 调试表达式触发的，并非业务代码显式调用产生。[02:12:45]
- 📱 老师通过真机 Demo 仿写 Runtime 结构体并读取缓存数据，验证了 `arm64` 下初始长度、满桶扩容和八分之七扩容等规则。[02:39:58]
- 📝 课程最后强调，网上许多关于 `cache_t` 的资料并不完整，建议学员认真整理本节课内容并完成“cache 详解”作业。[02:44:19]

### 思考 
1. **[为什么 `cache_t` 里缓存的是 `SEL` 和 `IMP`？](https://bibigpt.co/search?q=%E4%B8%BA%E4%BB%80%E4%B9%88%20%60cache_t%60%20%E9%87%8C%E7%BC%93%E5%AD%98%E7%9A%84%E6%98%AF%20%60SEL%60%20%E5%92%8C%20%60IMP%60%EF%BC%9F)**
  - 因为 Objective-C 方法调用最终需要通过方法名 `SEL` 找到对应的方法实现地址 `IMP`。`cache_t::insert` 方法的参数正好包含 `SEL` 和 `IMP`，说明 Runtime 会把它们作为缓存条目存入 `bucket_t`，以便下次调用时快速查找。
- 2. **[为什么调用过的方法有时在缓存中找不到？](https://bibigpt.co/search?q=%E4%B8%BA%E4%BB%80%E4%B9%88%E8%B0%83%E7%94%A8%E8%BF%87%E7%9A%84%E6%96%B9%E6%B3%95%E6%9C%89%E6%97%B6%E5%9C%A8%E7%BC%93%E5%AD%98%E4%B8%AD%E6%89%BE%E4%B8%8D%E5%88%B0%EF%BC%9F)**
  - 主要原因是缓存扩容时旧桶会被释放，旧缓存条目不会全部迁移到新桶中。比如在特定架构和桶长度下，某个新方法触发扩容后，之前缓存的条目可能就被清空，因此 LLDB 中看不到它。
- 3. **[`arm64` 和 `x86_64` 的缓存扩容规则有什么区别？](https://bibigpt.co/search?q=%60arm64%60%20%E5%92%8C%20%60x86_64%60%20%E7%9A%84%E7%BC%93%E5%AD%98%E6%89%A9%E5%AE%B9%E8%A7%84%E5%88%99%E6%9C%89%E4%BB%80%E4%B9%88%E5%8C%BA%E5%88%AB%EF%BC%9F)**
  - `arm64` 初始桶长度通常为 2，并且桶长度小于等于 8 时会等到桶满再扩容，之后按大于八分之七阈值扩容；`x86_64` 初始桶长度通常为 4，并按照四分之三阈值触发两倍扩容。


### 术语解释
- **`objc_class`**: Objective-C Runtime 中描述类对象的核心结构体，包含 `isa`、`superclass`、`cache_t`、`bits` 等信息。
- **`cache_t`**: 类对象中的方法缓存结构，用于缓存已经查找过的方法，提升后续消息发送时的方法查找效率。
- **`bucket_t`**: `cache_t` 内部真正存储缓存条目的桶结构，每个桶通常保存一组 `SEL` 和 `IMP`。
- **`SEL`**: 方法选择器，也可以理解为方法名，用于在 Runtime 中标识一个 Objective-C 方法。
- **`IMP`**: 方法实现指针，指向具体函数实现的地址，Runtime 找到 `IMP` 后即可执行对应方法逻辑。
